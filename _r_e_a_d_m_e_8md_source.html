16 ## Introduction
17 
18 This project aims to provide easy to use, platform independent and versatile module for implementing interactive command line for bare metal (or RTOS driven) embedded systems.
19 
20 See the [complete Doxygen documentation](https://hornmich.github.io/cmd_line/index.html "Complete Doxygen documentation").
21 
22 See the [example for Nucleo-F103RB board](https://github.com/hornmich/cmd_line_nucleo_ex "Nucleo-F103RB example") with FreeRTOS and STM32 CubeMX generated HAL
23 
24 ## Features
25 
26 * Multiple instances
27 * Embedded Help for the commands
28 * Commands with parameters
29 * Easy user commands definition
30 * Supports code reuse by nesting lists of commands
31 * Hardware abstraction - easy porting
32 * Provides IO tools for your commands

41 ## Quick start
42 
43 To start using the Command line module just:
44 
45 * clone the repository (preferably as a submodule) into your project,
46 * add
47 ``` C
48  #include "cmdline.h"
49 ```
50 in your source code where you plan to use the module, for example the `main.c`.
51 * Define your commands
52 * Fill in the initialization parameters for the instance configuration
53 * Create your command line instance
54 * Call periodically the main command line routine from RTOS task or in the main application cycle.
55 
56 ### Configuration
57 
58 The Command line modules configuration is separated into global configuration, that applies for all instances and to instance configuration, affecting just one command line instance.
59 
60 #### Global configuration
61 
62 The global configuration is implemented as a set of C preprocessor macros, defined in `cmdline.h` file, in Doxygen group `Cmd_Line_Lib_Config`.
63 
64 There are three macros defined. You will most of the time be fine with the default values. However if you are looking for memory footprint optimization, this is a good place to start.
65 
66 * `CMD_LINE_MAX_CMD_LEN` - Specifies the maximal number of characters for one line input, including command name and all parameters. This option affects the RW memory section usage.
67 * `CMD_LINE_MAX_INDENT` - Specifies the maximal indentation of the nested command lists. This option extensively affects the usage of the stack.
68 * `CMD_LINES_MAX_CNT` - Specifies the maximal number of Command line instances. E.g. if you want to have two separated command lines with different set of commands for UART1 and UART2 interfaces. This option affects heavily the use of RW memory section.
69 
70 #### Instance configuration
71 
72 The instance specific configuration is encapsulated by a `cmd_line_init_t` structure. In order to configure you instance, you need to define and fill in an instance of this structure and pass it to a `cmd_line_init`.
73 
74 The structer consists of the following parameters:
75 * `prompt` - Pointer to a string that will be used as a prompt. E.G "-->". Can be NULL.
76 * `rx_fn` - Pointer to a wrapper function for blocking character reception. This function has to be define by you and pointed to by this parameter.
77 * `tx_tc_fn` - Pointer to a wrapper function for blocking data transmit function. This function has to be define by you and pointed to by this parameter.
78 * `tx_nb_fn` - Pointer to a wrapper function for not blocking data transmit function. This function has to be define by you and pointed to by this parameter.
79 * `tx_buf` - Pointer to a buffer for storing data to be transmitted. This buffer has to be define by you and pointed to by this parameter, e.g. `cmd_line_tx_buf[CMD_LINE_TX_BUF_LEN];`
80 * `tx_buf_len` - The size of the `tx_buf` buffer.
81 * `rx_buf` - Pointer to a buffer for storing received data. This buffer has to be define by you and pointed to by this parameter.
82 * `rx_buf_len` - The size of the `rx_buf` buffer, e.g. `cmd_line_rx_buf[CMD_LINE_RX_BUF_LEN];`
83 * `line_buf` - Buffer used internally to store one complete line for further processing. This buffer has to be define by you and pointed to by this parameter, e.g. `cmd_line_line_buf[CMD_LINE_LINE_BUF_LEN];`
84 * `line_buf_size` - The size of the `line_buf` buffer.
85 * `max_args_cnt` - Specifies maximal number of parameters for any command.
86 * `io_timeout_ms` - Specifies the timeout durationn for IO operations in miliseconds.
87 * `args_starts` - Buffer used internally to point to arguments starts. This buffer has to be define by you and pointed to by this parameter, e.g. `char* cmd_line_args_starts[CMD_LINE_MAX_ARGS_CNT];`
88 * `echo_enabled` - Enables or disables echo.
89 * `cmd_root_lis` - Pointer to root list of commands.
90 
91 After filling in all parameters, pass the structure to the init function, like in the following example:
92 ``` C
93  cmd_line_desc_ptr_t command_line;
94 
95  cmd_line_init_t cmd_line_init_data = {
96  .prompt = CMD_LINE_PROMPT,
97  .tx_tc_fn = uart_tx_tc,
98  .tx_nb_fn = uart_tx_nb,
99  .rx_fn = uart_rx_tc,
100  .tx_buf = cmd_line_tx_buf,
101  .tx_buf_len = CMD_LINE_TX_BUF_LEN,
102  .rx_buf = cmd_line_rx_buf,
103  .rx_buf_len = CMD_LINE_RX_BUF_LEN,
104  .line_buf = cmd_line_line_buf,
105  .line_buf_size = CMD_LINE_LINE_BUF_LEN,
106  .max_args_cnt = CMD_LINE_MAX_ARGS_CNT,
107  .io_timeout_ms = CMD_LINE_IO_TIMEOUT_MS,
108  .args_starts = cmd_line_args_starts,
109  .echo_enabled = CMD_LINE_ECHO_ENABLED,
110  .cmd_root_lis = nucleo_cmd_list,
111  };
112 
113  command_line = cmd_line_init(&cmd_line_init_data);
114  if (command_line == NULL) {
115  Error_Handler();
116  }
117 ```

119 ### Define wrappers
120 
121 To make the module as portable as possible, there is an abstraction layer for the IO functions, called wrapper functions.
122 
123 This layer consists of 3 function pointers in the initialization structure with the following signature:
124 
125  * `int (*uart_tx_nb_t)(const char*, int len)`
126  * `int (*uart_tx_tc_t)(const char*, int len, int timeout_ms)`
127  * `int (*uart_rx_tc_t)(cmd_line_desc_ptr_t cmd_line_desc, char*, int len, int timeout_ms)`
128 
129  In your code you therefore need to define the functions with that signature and in the body put your hardware specific function calls. See the example below of one concrete implementation:
130 
131  ``` C
132 void uart_rx_cb(char c) {
133 
134 }
135 
136 int uart_tx_nb(const char* str, int len) {
137  if (HAL_UART_Transmit_DMA(&huart2, (uint8_t*)str, len) != HAL_OK) {
138  return CMD_LINE_ERR_GENERAL;
139  }
140  return CMD_LINE_SUCCESS;
141 }
142 
143 int uart_tx_tc(const char* str, int len, int timeout_ms) {
144  if (HAL_UART_Transmit(&huart2, (uint8_t*)str, len, timeout_ms) != HAL_OK) {
145  return CMD_LINE_ERR_GENERAL;
146  }
147  return CMD_LINE_SUCCESS;
148 }
149 
150 int uart_rx_tc(cmd_line_desc_ptr_t cmd_line_desc, char* str, int len, int timeout_ms) {
151  if (HAL_UART_Receive(&huart2, (uint8_t*)str, len, timeout_ms) != HAL_OK) {
152  return CMD_LINE_ERR_GENERAL;
153  }
154  return CMD_LINE_SUCCESS;
155 }
156  ```

158 ### Define commands
159 
160 Commands are grouped into lists. Depending on your global configuration macro `CMD_LINE_MAX_INDENT`, the list can contain other sublists or direct commands.
161 
162 #### Root list
163 
164 Each command line instance is linked with a root list of commands. Default root list is defined in `cmd.c` and `cmd.h` files as `basic_cmd_list`. Feel free to use and modify this root list as needed, for example by inserting you own set of sommands as sublists.
165 
166 #### Sublist
167 
168 You can use any command list as a sublist by using `CMD_LINE_INCLUDE_SUBLIST(sublist )` macro.
169 
170 E.g. inclusion of your own list of commands into the default root commands would look like this:
171 
172 ``` C
173 const cmd_desc_t* basic_cmd_list[] =
174  {
175  &cmd_desc_help,
176  &cmd_desc_version,
177  CMD_LINE_INCLUDE_SUBLIST(mylist),
178  CMD_LINE_END_OF_LIST
179  };
180 ```
181 
182 Using of sublist feature can help you in better code organization and code reuse between multiple command line instances.
183 
184 #### Command
185 
186 Before you will define your own lists of commands, it is important to understand forst how to define a command.
187 
188 Every command is defined by a function and by metadata. The command function has the following notation:
189 
190 ``` C
191  int (*cmd_do_fcn_t)(int argc, char** argv, cmd_line_desc_ptr_t cmd_line_desc)
192 ```
193 
194 The body of the function is the place where to put the code that will be executed when the command is invoked.
195 
196 The metadata are a structure with the fields:
197 * `name` - The name of the command without any white characters. The name is used to invoke the command.
198 * `one_line` - A brief description of the command.
199 * `description` - A detailed description of the command, including parameters and expected output.
200 * `cmd_fcn` - Pointer to the command function, described above.
201 
202 The command list is actually an array of pointers to descriptors. Therefore creating a list is done simply by defining the array of `cmd_desc_t` pointers.
203 
204 #### Example
205 See the example of LED control commands list.
206 
207 The header file with external declaration of the array of descriptors:
208 
209 ``` C
210 #ifndef LED_CMD_H_
211 #define LED_CMD_H_
212 
213 #include "cmdline.h"
214 
215 extern const cmd_desc_t* led_cmd_list[];
216 
217 #endif /* LED_CMD_H_ */
218 ```
219 
220 The source file with the command function bodies, the metadata filled and assigned to the array declared in the header file.
221 
222 ``` C
223 #include "led_cmd.h"
224 #include "stm32f1xx_hal.h"
225 
226 int cmd_do_ledon(int argc, char** argv, cmd_line_desc_ptr_t cmd_line_desc) {
227  HAL_GPIO_WritePin(LD2_GPIO_Port, LD2_Pin, GPIO_PIN_SET);
228  cmd_line_printf_tk(cmd_line_desc, "LED ON\r\n");
229  return CMD_LINE_SUCCESS;
230 }
231 
232 int cmd_do_ledoff(int argc, char** argv, cmd_line_desc_ptr_t cmd_line_desc) {
233  HAL_GPIO_WritePin(LD2_GPIO_Port, LD2_Pin, GPIO_PIN_RESET);
234  cmd_line_printf_tk(cmd_line_desc, "LED OFF\r\n");
235  return CMD_LINE_SUCCESS;
236 }
237 
238 int cmd_do_led(int argc, char** argv, cmd_line_desc_ptr_t cmd_line_desc) {
239  if (argc == 0) {
240  GPIO_PinState led_state = HAL_GPIO_ReadPin(LD2_GPIO_Port, LD2_Pin);
241  if (led_state == GPIO_PIN_SET) {
242  cmd_line_printf_tk(cmd_line_desc, "LED is ON\r\n");
243  }
244  else {
245  cmd_line_printf_tk(cmd_line_desc, "LED is OFF\r\n");
246  }
247  }
248 
249  else {
250  switch (argv[1][0]) {
251  case '0':
252  HAL_GPIO_WritePin(LD2_GPIO_Port, LD2_Pin, GPIO_PIN_RESET);
253  cmd_line_printf_tk(cmd_line_desc, "LED OFF\r\n");
254  break;
255  case '1':
256  HAL_GPIO_WritePin(LD2_GPIO_Port, LD2_Pin, GPIO_PIN_SET);
257  cmd_line_printf_tk(cmd_line_desc, "LED ON\r\n");
258  break;
259  default:
260  cmd_line_printf_tk(cmd_line_desc, "Invalid parameter.\r\n");
261  break;
262  }
263  }
264 
265  return CMD_LINE_SUCCESS;
266 }

340 ## Default commands
341 
342 The module is shipped with a very basic set of commands that alows you to start righ now and evaluate the function of your configuration (wrapers, buffers, UART periphery, RTOS, etc.)
343 
344 This basic set contais the following commands:
345 * `help` - If used standalone, prints the list of all commands with their brief description. If used with another commands names as parameters, prints detailes description for each one of them.
346 * `version` - Prints out the firmware version.
347 
348 ## IO Functions
349 
350 The module defines a set of IO functions that may be usefull in your commands to print texts to the terminal or to get user input:
351 
352 * `cmd_line_printf` - Not blocking C printf like function.
353 * `cmd_line_printf_tk` - Blocking C printf like function.
354 * `cmd_line_getchar` - Not blocking C like getchar function.
355 * `cmd_line_getchar_tk` - Blocking C like getchar function.
356 
357 Feel free to use these utility functions for you help.
